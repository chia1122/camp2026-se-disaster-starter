# Phase 0 Observations

## 我們一開始以為

- 只要把原始資訊列成卡片，就足以讓下一位協作者理解狀況。
- 實際整理後發現，Phase 0 的重點不是展示資料，而是展示「為什麼還不能相信或不能派工」。

## 我們偷偷做出的假設

- 看到「需要十幾個人」時，容易假設它已經是一個志工任務。
- 看到「官方公告」來源時，容易假設截圖日期與發布單位都可靠。
- 看到「集合點」時，容易假設該地點目前安全。

## 我們發現不清楚的地方

- `M-001` 只有「老雜貨店後面」，缺少可定位地址與現場窗口。
- `M-005` 缺飲用水的物資站可能是北側入口，也可能是河堤旁。
- `M-006` 道路封閉截圖不知道是哪一天，也不知道是否為官方公告。
- `M-007` 醫療需求缺少姓名、聯絡方式與橋梁位置，且涉及個資保護。
- `M-010` 「A 區先不要再派人」沒有說明是安全風險、人力過量，還是任務完成。

## 哪些資料看似事實但其實未確認

- `M-002` 溪畔活動中心早上有雨鞋，不代表下午仍有庫存。
- `M-004` 社群貼文說活動中心有很多雨鞋，但來源和時間都未確認。
- `M-009` 水電工班名單可能是昨天的，不能視為今日可用人力。

## 哪些欄位可能需要後續 schema 設計

- 原始來源類型與可信度。
- 查核狀態與人工確認點。
- 地點精確度或模糊地點說明。
- 是否能轉成任務，以及不能轉任務的原因。
- 資訊版本、更新時間與是否可能過期。
- 衝突資訊的關聯記錄。

## AI 幫助了什麼

- 協助把 Phase 0 首頁改成左右分欄工作台。
- 協助為每筆 messy report 標示可能分類、風險程度、不確定資訊與不能直接派工原因。
- 協助提醒 `events/**` 與 Phase 0 dirty data 不應被當成 normalized internal data。

## AI 可能誤導了什麼

- AI 容易把模糊描述補成看似完整的救災判斷，因此畫面必須標示「分類是協作輔助，不代表已確認事實」。
- AI 不能決定哪個災情為真，也不能決定救災或志工優先順序。

## 進入 SDD-lite 前，我們最想釐清的 3 件事

1. 哪些資訊欄位是不同 task line 都需要的共同欄位？
2. 「不能直接變成任務」應該是 UI 狀態、schema 欄位，還是 adapter 產物？
3. 衝突資訊要如何被保留與呈現，而不是被單一新資料覆蓋？

## 復盤後新增觀察

- Phase 0 應保留混亂與不確定性，避免過早整理成乾淨模型。
