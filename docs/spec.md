# Component Spec

## Problem statement

Phase 0 協作者需要在混亂災害資訊中快速看出：每筆資訊可能屬於哪一類、還缺哪些判斷條件、為什麼不能直接變成志工任務，以及如果要先建立工作草稿，應該保留哪些不確定性。

## Users and roles

- 原始資訊提供者：社群貼文、志工回報、電話轉述或現場回報的來源。
- 實際操作者：整理工作台的學生或協作者。
- 查核者：需要確認來源、地點、時間、狀態與安全性的人工 reviewer。
- 資訊接收者：下一位接手 UI、schema 或 handoff 的協作者。

## In scope

- 顯示所有 Phase 0 原始資訊。
- 對每筆資訊標示可能分類：通報、地點狀態、志工任務、人員指派。
- 用顏色標示風險或資訊狀態程度。
- 顯示人數預估，並允許依人數預估排序。
- 讓使用者以按鈕選填「還不知道如何判斷」與「不能直接變成任務」原因。
- 提供新增工作草稿頁面，草稿只存在前端 runtime state。
- 允許依風險程度、分類、查核狀態、人數預估、更新時間排序 Phase 0 工作台。

## Out of scope

- 不做後端、資料庫、登入、權限或 localStorage。
- 不自動判定災情真偽。
- 不把 Phase 0 dirty data 寫入 normalized shared fixtures。
- 不把使用者新增的草稿宣稱為 verified task。
- 不修改 `CommonRecord`。

## Main user flow

1. 使用者進入首頁的「第一階段原始資訊」。
2. 使用者查看 baby blue 背景中的分類工作台。
3. 使用者用排序選單依風險、分類、查核狀態、人數或時間整理顯示順序。
4. 使用者在每筆資訊中點選仍需確認的原因按鈕。
5. 使用者進入「新增工作」頁面，建立待人工確認的工作草稿。
6. 新增的草稿顯示在同頁右側，包含分類、風險、查核狀態與人數預估。

## States

- `needs_review`：需要人工確認，不可直接派工。
- `unverified`：來源未查核，不可當成已確認事實。
- 草稿工作：前端 runtime state，重新整理後會消失。
- 空草稿列表：顯示尚未新增工作草稿。

## 完成條件（Acceptance Criteria，簡稱 AC）

- [x] AC-01: 首頁顯示所有 Phase 0 原始資料，並標示分類與風險程度。
- [x] AC-02: 每筆資訊可用按鈕選填「還不知道如何判斷」與「不能直接變成任務」原因。
- [x] AC-03: 新增工作頁可輸入工作名稱、描述、分類、風險、查核狀態、來源與人數預估。
- [x] AC-04: 工作台可依分類、查核狀態、人數預估、風險程度與更新時間排序。

## Failure and uncertain cases

- 使用者若未填工作名稱或描述，表單不送出。
- 人數預估為 `0` 時，畫面顯示為「未估」。
- 選填原因只代表 reviewer 關注點，不代表該資訊已確認。
- 新增工作草稿不會進入 normalized fixture，也不會在重新整理後保存。

## Human confirmation points

- 模糊地點是否可定位。
- 來源是否可靠且時間是否仍有效。
- 人數預估是否足以派工。
- 醫療、個資或安全風險是否需要更嚴格的人工處理。
- 衝突資訊是否需要保留並交叉查核。
